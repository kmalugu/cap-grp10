PROJECT VIVA CONCEPTS
University AI Assistant

This file explains the main technical concepts used in the project in simple language for viva preparation.


1. Artificial Intelligence Assistant

What it means:
- An AI assistant is a system that takes user input, understands it, and gives a useful response.

How it is used here:
- The project acts as a university assistant.
- It answers questions about courses, fees, timetables, faculty, academic calendar, and university policies.

Important viva line:
- "This is an AI-based university assistant that combines structured data lookup, RAG, memory, and an LLM to answer student queries."


2. Large Language Model (LLM)

What it means:
- An LLM is a model trained on large amounts of text to understand and generate language.

How it is used here:
- The project uses Gemini through LangChain.
- The LLM generates the final natural-language answer.

Why it is useful:
- It can understand user questions written in normal language.
- It can summarize, explain, and generate human-like responses.

Limitation:
- LLMs can hallucinate, meaning they may generate answers that sound correct but are not actually based on real data.

Project solution:
- The project reduces hallucination by using RAG and by adding a disclaimer when the answer is not found in official documents.

Viva line:
- "The LLM is used for response generation, but we improve reliability by grounding it with retrieved documents and memory."


3. Retrieval-Augmented Generation (RAG)

What it means:
- RAG is a technique where the system first retrieves relevant documents and then gives them to the LLM to generate an answer.

Simple idea:
- Retrieval finds useful information.
- Generation converts that information into a final answer.

How it works in this project:
1. University documents are stored in data/rag_docs/.
2. They are converted into chunks and embeddings.
3. Those embeddings are stored in FAISS.
4. When the user asks a question, similar chunks are retrieved.
5. The LLM uses those chunks to answer.

Why it is useful:
- It allows the assistant to answer from project-specific university documents.
- It improves factual grounding.

Viva line:
- "RAG helps the model answer based on our own university documents instead of relying only on general pretrained knowledge."


4. Document Loading

What it means:
- Before using RAG, documents must be read from files and converted into machine-processable text.

How it is used here:
- The project loads:
  - .txt files
  - .pdf files
- This is handled by utils/file_loader.py.

Why it matters:
- The system cannot retrieve information unless the source documents are first loaded properly.


5. Chunking

What it means:
- Chunking means splitting a large document into smaller pieces.

Why chunking is needed:
- LLMs and vector search work better on smaller text units.
- If an entire PDF is stored as one block, retrieval becomes inefficient and less accurate.

How it is used here:
- utils/text_splitter.py uses RecursiveCharacterTextSplitter.
- It splits documents into chunks of:
  - chunk_size = 500
  - chunk_overlap = 50

Meaning of overlap:
- Overlap means some text is repeated between consecutive chunks.
- This helps preserve context across chunk boundaries.

Example:
- If one chunk ends in the middle of an important explanation, overlap helps the next chunk still contain some of that meaning.

Viva line:
- "We use chunking to divide long documents into smaller searchable units, and overlap helps preserve context between adjacent chunks."


6. Embeddings

What it means:
- Embeddings are numerical vector representations of text.
- They capture semantic meaning.

Simple intuition:
- Similar sentences get similar vectors.

How it is used here:
- The project uses HuggingFace sentence-transformer embeddings.
- The embedding model name is read from utils/config.py.

Why embeddings matter:
- Keyword matching only checks exact words.
- Embeddings allow semantic search.
- So a question like "hostel rules" can match a document that says "residential guidelines."

Viva line:
- "Embeddings convert text into vectors so that semantically similar text stays close in vector space."


7. Vector

What it means:
- A vector is an ordered list of numbers.
- In AI, text, images, or audio can be represented as vectors.

How it is used here:
- Every chunk of text becomes a vector.
- User queries also become vectors.
- Similarity search compares these vectors.


8. Semantic Search

What it means:
- Semantic search finds relevant text based on meaning, not only exact words.

Difference from keyword search:
- Keyword search looks for exact terms.
- Semantic search looks for concept similarity.

Project use:
- The retriever compares the query embedding with stored document embeddings.

Why it helps:
- Users may ask the same question in many ways.


9. Vector Database / Vector Store

What it means:
- A vector database stores embeddings and supports similarity search.

How it is used here:
- FAISS is used for RAG document retrieval.
- ChromaDB is used for long-term memory storage.

Why two systems are used:
- FAISS is used mainly for document retrieval.
- ChromaDB is used for student-specific memory items like interests and doubts.

Viva line:
- "We use FAISS for document retrieval and ChromaDB for persistent vector memory."


10. FAISS

Full form:
- Facebook AI Similarity Search

What it means:
- FAISS is a library for efficient similarity search on vectors.

How it is used here:
- Document chunk embeddings are stored in FAISS.
- At runtime, the retriever loads the FAISS index and searches top-k relevant chunks.

Why it is chosen:
- Fast vector similarity search
- Good for retrieval pipelines


11. ChromaDB

What it means:
- ChromaDB is a vector database used for persistent semantic memory.

How it is used here:
- It stores student-specific long-term memories such as:
  - doubts
  - interests
  - electives

Why it is useful:
- Later, the system can search previous student memories semantically.
- This makes the assistant more personalized.


12. Similarity Search

What it means:
- Similarity search finds items whose vectors are closest to a query vector.

How it works:
1. Convert query to embedding
2. Compare with stored vectors
3. Return nearest matches

In simple words:
- It answers: "Which stored text is most similar in meaning to this question?"


13. Top-k Retrieval

What it means:
- Top-k means returning the best k matches.

How it is used here:
- The retriever often returns top 3 or top 5 documents.

Why it matters:
- Too few results may miss useful context.
- Too many results may add noise.

Viva line:
- "Top-k retrieval balances relevance and context size."


14. Prompt Engineering

What it means:
- Prompt engineering means designing instructions given to the LLM so it behaves as needed.

How it is used here:
- If documents are found:
  - the prompt tells the model to answer only from documentation
- If documents are not found:
  - the prompt tells the model to answer carefully with a disclaimer
- In memory extraction:
  - prompts ask the LLM to return JSON
  - prompts ask for summary, topics, and extracted student details

Why it matters:
- The model's output quality depends strongly on the prompt.


15. Context

What it means:
- Context is the information given to the model along with the user query.

Types of context in this project:
- current user question
- student memory
- retrieved documents
- conversation history

Why context matters:
- Better context gives more accurate and relevant responses.


16. Guardrails

What it means:
- Guardrails are safety rules that prevent harmful or disallowed outputs.

How it is used here:
- backend/guardrails.py checks the user's message before processing.
- It blocks content related to:
  - violence
  - illegal activity
  - academic dishonesty
  - harassment
  - privacy violation
  - suspicious behavior

Why it is useful:
- It keeps the assistant aligned with responsible AI behavior.

Viva line:
- "Guardrails act as the first safety layer before any retrieval or generation happens."


17. Caching

What it means:
- Caching means storing results temporarily so repeated work can be avoided.

How it is used here:
- backend/services/cache.py stores recent responses.
- The LLM object is cached.
- The FAISS vector store is cached.
- The embedding model is cached.

Why it helps:
- Faster responses
- Less repeated computation
- Lower resource usage

Viva line:
- "Caching improves performance by reusing previously computed responses and loaded resources."


18. Memory in AI systems

What it means:
- Memory lets the assistant remember useful information beyond the current message.

Why it matters:
- Without memory, every conversation starts from zero.
- With memory, the assistant can personalize responses.

This project uses three memory types:
- entity memory
- summary memory
- vector memory


19. Entity Memory

What it means:

Examples:

- program
- year

How it is stored:

22. LangGraph

What it means:
- LangGraph is a framework for building step-by-step AI workflows as graphs.

How it is used here:
- memory_graph.py creates a workflow for memory extraction.
- The nodes are:
  1. extract entities
  2. identify topics
  3. summarize conversation
  4. extract vector memories
  5. persist memory

Why it is useful:
- It organizes a multi-step AI workflow clearly.
- It is better than writing one large monolithic function.

Viva line:
- "LangGraph is used to structure the memory extraction pipeline into separate sequential nodes."

61. Likely viva questions with short answers

Q. Why did you use RAG?
- To ground the LLM in university-specific documents and reduce hallucination.

Q. Why do you need chunking?
- Large documents are hard to search directly, so we split them into smaller meaningful units for better retrieval.

Q. What are embeddings?
- Numerical vector representations of text that capture semantic meaning.

Q. Why use FAISS?
- It provides efficient similarity search for document embeddings.

Q. Why use ChromaDB also?
- FAISS handles document retrieval, while ChromaDB stores persistent student memory for personalization.

Q. What is the role of LangGraph?
- It structures the memory extraction workflow into multiple clear steps.

Q. How do you reduce hallucination?
- By retrieving official documents, prompting the model to answer only from them, and using disclaimers when no docs are found.

Q. Why use structured tools instead of only LLM?
- Structured data lookups are faster and more accurate for factual tabular information.

Q. What is the role of guardrails?
- They block unsafe or policy-violating prompts before generation.

Q. What is the role of memory?
- It helps the assistant personalize responses and remember important student information across conversations.

Q. What is the difference between entity memory and vector memory?
- Entity memory stores fixed structured profile data, while vector memory stores semantic long-term information for similarity search.

Q. Why use FastAPI and Streamlit?
- FastAPI provides a clean backend API, and Streamlit gives a simple Python-based frontend.


62. Final viva revision paragraph

This project is a hybrid AI university assistant. It uses FastAPI as the backend and Streamlit as the frontend. For direct factual queries, it uses structured CSV and JSON data. For document-based answering, it uses RAG, where documents are loaded, chunked, embedded, indexed in FAISS, and retrieved during question answering. The final response is generated by Gemini. To make the system personalized, it stores entity memory, summary memory, and vector memory, with ChromaDB used for long-term semantic memory. To keep the system safe, guardrails filter harmful or dishonest prompts. Caching improves speed, and evaluation scripts help measure retrieval quality, response quality, and latency.
